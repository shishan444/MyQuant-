# 05 - 状态管理与服务层

## 1. 概述

MyQuant 前端采用双层状态管理架构:

- **Zustand**: 管理客户端 UI 状态 (侧边栏、图表配置、实验室参数)
- **React Query (TanStack Query)**: 管理服务端数据状态 (API 请求、缓存、同步)

服务层基于 Axios 封装, 按 domain 拆分为独立模块。

### 3.5 useEvolution

文件位置: `src/hooks/useEvolution.ts`

Query Key 命名空间:

```typescript
evolutionKeys = {
  all: ["evolution"],
  tasks: (filters?) => ["evolution", "tasks", filters],
  task: (id) => ["evolution", "task", id],
  history: (id) => ["evolution", "history", id],
  strategies: (id) => ["evolution", "strategies", id],
}
```

Hooks:

| Hook                    | 类型     | 说明                               |
|------------------------|---------|------------------------------------|
| useEvolutionTasks      | query   | 任务列表, 支持 status 过滤           |
| useEvolutionTask       | query   | 单个任务详情, enabled=!!id          |
| useEvolutionHistory    | query   | 代际历史记录, enabled=!!id          |
| useEvolutionStrategies | query   | 任务的策略列表, enabled=!!id        |
| useCreateEvolutionTask | mutation| 创建任务, 成功后刷新任务列表         |
| useStopEvolutionTask   | mutation| 停止任务, 刷新任务+详情缓存          |
| usePauseEvolutionTask  | mutation| 暂停任务, 刷新任务+详情缓存          |
| useResumeEvolutionTask | mutation| 恢复任务, 刷新任务+详情缓存          |
| useEvolutionWebSocket  | effect  | WebSocket 订阅, 实时更新缓存         |

Mutation 成功后的缓存失效策略:

- `createTask`: `invalidateQueries(["evolution", "tasks"])`
- `stopTask`: `invalidateQueries(["evolution", "tasks"])` + `invalidateQueries(evolutionKeys.task(taskId))`
- `pauseTask` / `resumeTask`: 同 stopTask

### 3.6 WebSocket 实时更新

`useEvolutionWebSocket` 详细行为:

连接参数:

- URL: `ws(s)://{host}/ws/evolution/{taskId}`
- Host 来源: `VITE_WS_URL` 环境变量 > `window.location.host`

消息处理:

```
收到 WebSocket 消息
    |
    v
JSON.parse -> GenerationUpdate
    |
    +-- type === "generation_complete" | "evolution_complete"
    |     |
    |     v
    |   setQueryData(evolutionKeys.task(taskId), 合并更新)
    |     更新字段: current_generation, best_score, champion_dna, status, target_score, max_generations
    |
    +-- 其他类型
          |
          v
        setQueryData(evolutionKeys.task(taskId), 展开合并)
    |
    v
scheduleInvalidation (2s debounce)
    |
    v
invalidateQueries(evolutionKeys.history(taskId))
invalidateQueries(evolutionKeys.strategies(taskId))
```

断线重连: `setTimeout(connect, 3000)` 在 `onclose` 回调中触发。

清理: 组件卸载时关闭 WebSocket 连接, 清理重连定时器和失效定时器。

## 4. 服务层 (API Client)

### 4.1 api.ts (Axios 实例)

文件位置: `src/services/api.ts`

```typescript
const api = axios.create({
  baseURL: import.meta.env.VITE_API_BASE_URL || "",
  timeout: 30000,
  headers: { "Content-Type": "application/json" },
});
```

请求转换:

- FormData: 删除 Content-Type, 让浏览器自动设置 multipart boundary
- JSON 对象: 手动 JSON.stringify, 设置 Content-Type: application/json

响应拦截器:

```typescript
api.interceptors.response.use(
  (response) => response,
  (error) => {
    const message =
      error.response?.data?.detail || error.message || "请求失败";
    return Promise.reject(new Error(message));
  }
);
```

错误信息优先级: `response.data.detail` > `error.message` > "请求失败"

## 6. 数据流总结

### 6.1 客户端状态流

```
用户操作 (点击折叠按钮)
    |
    v
Zustand action (toggleSidebar)
    |
    v
状态更新 (sidebarCollapsed: true)
    |
    v
React 重渲染 (Sidebar 组件)
```

### 6.2 服务端数据流

```
组件挂载
    |
    v
React Query hook (useQuery)
    |
    v
Service 函数 (api.get)
    |
    v
Axios 实例 (拦截器处理)
    |
    v
HTTP 请求 -> 后端 API
    |
    v
响应拦截 (错误处理)
    |
    v
React Query 缓存
    |
    v
组件重渲染
```

### 6.3 实时数据流

```
WebSocket 推送
    |
    v
useEvolutionWebSocket
    |
    v
setQueryData (直接更新缓存, 无网络请求)
    |
    v (2s debounce)
invalidateQueries (触发后台重新获取)
    |
    v
UI 自动更新
```

## 7. 设计约束

1. Zustand 仅管理纯 UI 状态, 不缓存服务端数据
2. React Query 作为唯一的服务端数据管理方案, 不使用 useState 存储 API 数据
3. Query Key 按层级命名空间组织, 确保精确失效
4. 所有 mutation 的 onSuccess 回调中执行 invalidateQueries, 确保列表数据同步
5. WebSocket 更新采用 setQueryData + 延迟 invalidate 的组合策略, 兼顾实时性和一致性
6. Axios 实例统一处理错误提取, Hooks 层统一处理 toast 通知
7. 文件上传 (CSV) 使用 FormData, 超时时间延长至 120s/300s
8. API base URL 通过环境变量 `VITE_API_BASE_URL` 配置, 默认为空 (同源)
