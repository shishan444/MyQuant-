# 03 - 数据库设计

## 1. 概述

MyQuant 使用 SQLite 作为持久化存储，启用 WAL (Write-Ahead Logging) 模式以支持读写并发。系统使用两个数据库文件：

- `data/quant.db`: 主数据库，存储策略、回测结果、数据集元数据、Schema 版本
- `data/evolution.db`: 进化专用数据库，存储进化任务、代际快照、进化历史

数据库初始化通过版本化迁移机制管理，迁移文件位于 `migrations/` 目录，由 `api/db_ext.py` 中的 `init_db_ext()` 函数执行。

---

## 2. Schema 版本管理

### 2.1 schema_version 表

记录已应用的迁移版本号，确保迁移幂等性。

| 列名         | 类型      | 约束            | 说明                |
|------------|---------|---------------|-------------------|
| version    | INTEGER | PRIMARY KEY   | 迁移版本号            |
| applied_at | TEXT    | NOT NULL      | 应用时间 (UTC ISO 格式) |

迁移版本记录：

| version | 说明                                      |
|---------|-----------------------------------------|
| 1       | 创建 schema_version 表                    |
| 2       | 创建 strategy 表                          |
| 3       | 创建 backtest_result 表                   |
| 4       | 创建 dataset_meta 表                      |
| 5       | ALTER evolution_task 添加扩展列             |
| 6       | ALTER evolution_task 添加 MTF 支持列        |
| 7       | ALTER evolution_task 添加约束条件列          |

## 3. 表结构详细设计

### 3.1 evolution_task -- 进化任务表

存储每次策略进化任务的配置和状态。一条记录代表一次完整的进化过程。

| 列名                 | 类型      | 约束                 | 默认值      | 说明                                                      |
|--------------------|---------|----------------------|----------|-----------------------------------------------------------|
| task_id            | TEXT    | PRIMARY KEY          | -        | 任务唯一标识 (UUID)                                          |
| status             | TEXT    | NOT NULL             | running  | 任务状态: pending / running / paused / stopped / completed |
| symbol             | TEXT    | NOT NULL             | -        | 交易对 (如 BTCUSDT)                                         |
| timeframe          | TEXT    | NOT NULL             | -        | 执行时间框架 (如 4h)                                        |
| target_score       | REAL    | NOT NULL             | -        | 目标分数 (0-100)，达到则提前停止                               |
| score_template     | TEXT    | NOT NULL             | -        | 评分模板: profit_first / steady / risk_first / custom       |
| initial_dna        | TEXT    | NOT NULL             | -        | 初始 DNA JSON (StrategyDNA.to_json())                      |
| champion_dna       | TEXT    | NULL                 | -        | 冠军 DNA JSON (进化完成后写入)                                |
| population_size    | INTEGER | -                    | 15       | 每代种群大小                                                 |
| max_generations    | INTEGER | -                    | 200      | 最大进化代数                                                 |
| elite_ratio        | REAL    | -                    | 0.5      | 精英保留比例 (0.0-1.0)                                      |
| n_workers          | INTEGER | -                    | 6        | 并行工作数 (预留)                                            |
| current_generation | INTEGER | -                    | 0        | 当前已完成的代数                                              |
| leverage           | INTEGER | -                    | 1        | 杠杆倍数 (1-10)                                             |
| direction          | TEXT    | -                    | long     | 交易方向: long / short / mixed                              |
| data_start         | TEXT    | NULL                 | -        | 数据起始时间 (用户指定)                                       |
| data_end           | TEXT    | NULL                 | -        | 数据结束时间 (用户指定)                                       |
| data_time_start    | TEXT    | NULL                 | -        | 实际数据最早时间 (从 Parquet 读取)                             |
| data_time_end      | TEXT    | NULL                 | -        | 实际数据最晚时间 (从 Parquet 读取)                             |
| data_row_count     | INTEGER | -                    | 0        | 数据行数                                                    |
| best_score         | REAL    | NULL                 | -        | 历史最高分数                                                 |
| indicator_pool     | TEXT    | NULL                 | -        | 指标池 JSON (限制进化可用的指标列表)                             |
| timeframe_pool     | TEXT    | NULL                 | -        | 时间框架池 JSON (MTF 模式下的可用时间框架列表)                   |
| mode               | TEXT    | NULL                 | -        | 进化模式 (预留扩展)                                          |
| stop_reason        | TEXT    | NULL                 | -        | 停止原因: target_reached / stagnation / decline / max_generations / error |
| created_at         | TEXT    | NOT NULL             | -        | 创建时间 (UTC ISO 格式)                                      |
| updated_at         | TEXT    | NOT NULL             | -        | 最后更新时间 (UTC ISO 格式)                                   |

**状态流转**:

```
pending -> running -> completed
                  -> stopped (用户手动停止)
                  -> stopped (error)
         -> paused -> running (恢复)
```

### 3.2 generation_snapshot -- 代际快照表

存储每代的完整种群信息，用于检查点恢复。

| 列名             | 类型      | 约束                       | 说明                                        |
|----------------|---------|--------------------------|---------------------------------------------|
| task_id        | TEXT    | NOT NULL                 | 关联的进化任务 ID                              |
| generation     | INTEGER | NOT NULL                 | 代数编号 (从 1 开始)                          |
| best_score     | REAL    | NOT NULL                 | 该代最高分数                                   |
| avg_score      | REAL    | NOT NULL                 | 该代平均分数                                   |
| best_dna       | TEXT    | NOT NULL                 | 该代最佳个体 DNA JSON                          |
| population_json| TEXT    | NOT NULL                 | 完整种群 JSON (所有个体的 DNA 序列化数组)         |
| mutation_log   | TEXT    | NULL                     | 变异诊断信息 JSON                              |
| wf_months      | TEXT    | NULL                     | Walk-Forward 月份 (预留)                      |
| created_at     | TEXT    | NOT NULL                 | 创建时间                                      |

**联合主键**: (task_id, generation)

**population_json 示例**:

```json
[
  {"strategy_id": "...", "signal_genes": [...], "logic_genes": {...}, ...},
  {"strategy_id": "...", "signal_genes": [...], "logic_genes": {...}, ...}
```

## 4. 表关系图

```
+------------------+       +-------------------+
| evolution_task   |       | generation_snapshot|
| (task_id PK)     |<----->| (task_id+gen PK)  |
+------------------+       +-------------------+
        |
        | 1:N
        v
+------------------+
| evolution_history|
| (task_id+gen PK) |
+------------------+

+------------------+       +-------------------+
| strategy         |       | backtest_result   |
| (strategy_id PK) |<----->| (result_id PK)    |
+------------------+  1:N  | FK strategy_id    |
                          +-------------------+

+------------------+
| dataset_meta     |
| (dataset_id PK)  |
+------------------+

+------------------+
| schema_version   |
| (version PK)     |
+------------------+
```

关系说明：

- `evolution_task` 1:N `generation_snapshot`: 一个任务有多个代际快照
- `evolution_task` 1:N `evolution_history`: 一个任务有多条历史记录
- `strategy` 1:N `backtest_result`: 一个策略可有多次回测结果，级联删除
- `dataset_meta` 独立表，无外键关联
- `schema_version` 独立表，用于迁移管理

## 5. 数据库配置

### 5.1 连接配置

```python
conn = sqlite3.connect(str(db_path))
conn.execute("PRAGMA journal_mode=WAL")    # 启用 WAL 模式
conn.row_factory = sqlite3.Row             # 返回字典式行
```

### 5.2 WAL 模式

WAL (Write-Ahead Logging) 模式的选择原因：

- 允许读操作与写操作并发（读不阻塞写，写不阻塞读）
- 适合单写多读的场景（EvolutionRunner 写入，API 路由读取）
- 性能优于默认的 DELETE 模式
- 无需额外配置，SQLite 内建支持

### 5.3 并发访问模式

```
EvolutionRunner (写):
  - INSERT/UPDATE evolution_task
  - INSERT generation_snapshot
  - INSERT evolution_history

FastAPI Routes (读):
  - SELECT evolution_task (任务列表/详情)
  - SELECT evolution_history (历史图表)
  - SELECT/INSERT/UPDATE strategy
  - SELECT/INSERT backtest_result
  - SELECT/INSERT/UPDATE dataset_meta

WAL 模式下，读写可并发进行，无锁竞争。
```

## 6. 数据生命周期

### 6.1 进化任务数据

```
创建任务:
  evolution_task (status='pending')

进化过程中:
  evolution_task (status='running', current_generation 更新)
  generation_snapshot (每代插入一条)
  evolution_history (每代插入一条)

进化完成:
  evolution_task (status='completed', champion_dna 写入)

清理策略:
  - generation_snapshot 数据量最大 (每代 ~200KB)
  - 建议任务完成后定期清理超过 N 代的 snapshot
  - evolution_history 保留用于历史分析，长期保留
```

### 6.2 回测结果数据

```
回测执行:
  backtest_result (插入一条)

数据增长:
  equity_curve JSON 和 trades_json JSON 可能较大
  单次回测结果约 10KB-500KB

清理策略:
  - 关联的策略删除时级联删除
  - 定期清理 run_source='comparison' 的临时对比结果
```

### 6.3 数据集元数据

```
数据导入:
  dataset_meta (UPSERT)
  Parquet 文件 (覆盖/合并)

更新触发:
  每次导入更新 row_count, time_start, time_end, file_size_bytes
  更新 last_import_at

清理策略:
  删除 dataset_meta 记录 + 对应 Parquet 文件
```

## 7. 迁移机制

### 7.1 迁移流程

```python
init_db_ext(db_path):
  1. init_db(db_path)  # 创建 evolution_task/generation_snapshot/evolution_history
  2. _ensure_schema_version_table()
  3. 获取已应用版本: _get_applied_versions()
  4. 遍历 migrations/*.sql:
     - 提取文件名前缀数字作为版本号
     - 跳过已应用的版本
     - 跳过版本 5/6 (通过 ALTER TABLE 处理)
     - 执行 SQL
     - 记录版本号
  5. _apply_alter_evolution_task()  # 添加扩展列
  6. _apply_mtf_columns()          # 添加 MTF 列
  7. _apply_constraint_columns()   # 添加约束条件列
```

### 7.2 幂等性保证

所有迁移操作设计为幂等：

- `CREATE TABLE IF NOT EXISTS`: 表不存在则创建
- `ALTER TABLE ADD COLUMN`: 先检查列是否存在，不存在才添加
- `INSERT OR IGNORE INTO schema_version`: 版本号已记录则跳过

### 7.3 迁移文件命名规范

```
migrations/
  001_create_schema_version.sql
  002_create_strategy.sql
  003_create_backtest_result.sql
  004_create_dataset_meta.sql
  005_alter_evolution_task.sql      (由 Python ALTER 逻辑替代)
  006_mtf_support.sql               (由 Python ALTER 逻辑替代)
```

文件名前缀三位数字表示版本号，按数字顺序执行。
