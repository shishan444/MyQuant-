# 01 - 系统架构与模块划分

## 1. 架构总览

MyQuant 采用前后端分离的单机部署架构，后端使用 Python FastAPI 提供 REST API 与 WebSocket 服务，前端使用 React TypeScript 构建单页应用。数据层使用 SQLite 持久化元数据、Parquet 文件存储行情数据。

```
+---------------------------------------------------------------+
|                        React SPA (TypeScript)                  |
|  Pages: Lab / Evolution / Strategies / Trading / Data / Settings|
+-------------------+-------------------+------------------------+
        | REST API          | WebSocket          |
        v                   v                    |
+---------------------------------------------------------------+
|                    FastAPI Application                         |
|  Routers: data / evolution / strategies / validate / ws / config|
|  Schema: Pydantic V2  |  Deps: db_path, data_dir              |
+-----------------------------+---------------------------------+
                              |
            +-----------------+------------------+
            |                                    |
    +-------v--------+                +----------v----------+
    |  EvolutionRunner|                |  Core Business Logic |
    |  (Daemon Thread)|                |  (Synchronous)      |
    +-------+--------+                +----------+----------+
            |                                    |
    +-------v------------------------------------v----------+
    |                    Data Layer                           |
    |   SQLite (WAL)          |        Parquet (PyArrow)     |
    +--------------------------------------------------------+
```

### 1.1 架构特征

| 特征           | 说明                                                         |
|--------------|------------------------------------------------------------|
| 部署模式       | 单机部署，前后端独立进程                                       |
| 后端框架       | Python 3.12 + FastAPI + uvicorn                              |
| 前端框架       | React 19 + TypeScript + Vite                                 |
| 数据库         | SQLite，启用 WAL 模式保证读写并发                               |
| 行情存储       | Parquet (PyArrow engine)，按 `{symbol}_{timeframe}.parquet` 命名 |
| 实时通信       | WebSocket，推送进化过程逐代更新                                 |
| 后台任务       | EvolutionRunner 守护线程，轮询 SQLite 获取待执行任务              |

### 1.2 进程模型

```
[uvicorn 主进程]
    |
    +-- FastAPI event loop (asyncio)
    |       |
    |       +-- REST API handlers (同步/异步路由)
    |       +-- WebSocket endpoint (ws/evolution/{task_id})
    |
    +-- EvolutionRunner (threading.Thread, daemon=True)
            |
            +-- 轮询 evolution_task 表 (poll_interval=2s)
            +-- 执行 EvolutionEngine.evolve() (CPU 密集)
            +-- 通过 asyncio.run_coroutine_threadsafe 推送 WS 消息
```

EvolutionRunner 运行在独立守护线程中，避免阻塞 FastAPI 事件循环。进化引擎评估个体时执行同步的回测计算（加载 Parquet、计算指标、生成信号、运行回测），这是 CPU 密集操作。通过 `asyncio.run_coroutine_threadsafe` 将 WS 推送调度回主线程的事件循环。

## 2. 后端模块划分

后端代码分为两层：`api/` 层负责 HTTP 接口与协议转换，`core/` 层承载业务逻辑。所有 `core/` 模块不依赖 FastAPI，可独立测试。

### 2.1 api/ -- HTTP 接口层

```
api/
  __init__.py
  __main__.py          # uvicorn 启动入口
  app.py               # FastAPI 应用工厂 (create_app)
  db_ext.py            # 扩展数据库层 (strategy/backtest_result/dataset_meta CRUD)
  deps.py              # 依赖注入 (get_db_path, get_data_dir)
  runner.py            # EvolutionRunner 守护线程
  schemas.py           # Pydantic V2 请求/响应模型
  routes/
    __init__.py
    chart_config.py    # 图表配置 API
    config.py          # 全局配置 API (symbols, timeframes)
    data.py            # 数据导入/管理 API
    evolution.py       # 进化任务 CRUD + 控制 API
    strategies.py      # 策略 CRUD + 回测 API
    validate.py        # 假设验证 API
    ws.py              # WebSocket 端点 (实时进化进度)
```

#### 2.1.1 app.py -- 应用工厂

`create_app(db_path, data_dir)` 负责创建 FastAPI 实例，配置 CORS、lifespan 事件、路由挂载。lifespan 阶段完成：

1. 初始化数据库（`init_db_ext`，执行迁移）
2. 将 `db_path` 和 `data_dir` 存入 `app.state` 供依赖注入
3. 创建并启动 EvolutionRunner 守护线程
4. 注册 WS 推送回调（Runner -> WebSocket Manager 桥接）

#### 2.1.2 schemas.py -- 请求/响应模型

使用 Pydantic V2 BaseModel，定义所有 API 的输入输出结构。核心模型包括：

- **DNAModel**: 策略基因组的完整序列化模型，包含 signal_genes、logic_genes、execution_genes、risk_genes 四大基因簇
- **EvolutionTaskCreate / EvolutionTaskResponse**: 进化任务的创建与查询
- **BacktestRequest / BacktestResponse**: 回测请求与结果
- **DatasetResponse / DataImportResponse**: 数据集元信息与导入结果

#### 2.1.3 runner.py -- 进化任务执行器

`EvolutionRunner` 是 `threading.Thread` 子类，以守护线程方式运行。核心职责：

1. 每 2 秒轮询 `evolution_task` 表中 status='pending' 的任务
2. 解析 initial_dna，构建 EvolutionEngine 实例
3. 通过 `evaluate_fn` 回调评估每个个体（加载 Parquet -> 计算指标 -> 生成信号 -> 回测 -> 评分）
4. 每代结束后：保存 snapshot/history 到数据库、更新 current_generation、推送 WS 消息
5. 检查 stop/pause 状态，支持中途停止

#### 2.1.4 routes/ -- 路由模块

| 路由文件         | 路径前缀           | 职责                                  |
|--------------|----------------|---------------------------------------|
| config.py    | /api/config    | 获取可用 symbols、timeframes、数据源列表   |
| data.py      | /api/data      | CSV 导入、数据集列表/预览/删除、OHLCV 查询 |
| evolution.py | /api/evolution | 进化任务 CRUD、启动/停止/暂停、历史查询     |
| strategies.py| /api/strategies| 策略 CRUD、单策略回测、多策略对比回测       |
| validate.py  | /api/validate  | WHEN/THEN 假设验证                     |
| ws.py        | /ws/evolution/{task_id} | WebSocket 实时推送进化进度      |
| chart_config.py | /api/chart    | 图表指标配置                            |

### 2.2 core/ -- 业务逻辑层

#### 2.2.1 core/strategy -- 策略基因组

```
core/strategy/
  dna.py         # StrategyDNA 数据结构定义
  executor.py    # dna_to_signals 信号转换引擎
  validator.py   # DNA 结构验证器
```

**dna.py** 定义了策略的基因表示，是整个系统的核心数据结构：

- **SignalGene**: 单个信号条件，由 indicator + params + role + condition 组成
- **LogicGenes**: 信号组合逻辑（AND/OR），分 entry_logic 和 exit_logic
- **ExecutionGenes**: 执行层参数（timeframe、symbol）
- **RiskGenes**: 风控参数（stop_loss、take_profit、position_size、leverage、direction）
- **TimeframeLayer**: 多时间框架层，每层包含独立的 signal_genes 和 logic_genes
- **StrategyDNA**: 完整策略基因组，支持单时间框架和多时间框架（MTF）两种模式

**executor.py** 是信号转换的关键桥梁模块，负责将 StrategyDNA 转换为 vectorbt 可用的 entry/exit 布尔序列：

- `evaluate_condition()`: 将条件字典转换为布尔 Series，支持 16 种条件类型（lt/gt/le/ge、cross_above/cross_below、price_above/price_below、cross_above_series/cross_below_series、lookback_any/lookback_all、touch_bounce/role_reversal/wick_touch）
- `_get_indicator_column()`: 从增强 DataFrame 中解析指标列名
- `dna_to_signals()`: 完整的 DNA -> 信号转换，支持单时间框架和 MTF 模式
- `evaluate_layer()`: 评估单个时间框架层
- `resample_signals()`: 高时间框架信号向低时间框架前向填充

**validator.py** 验证 DNA 结构的合法性（至少 1 个 entry_trigger + 1 个 exit_trigger，参数范围合法等）。

#### 2.2.2 core/features -- 指标计算

```
core/features/
  indicators.py    # 指标计算引擎 + compute_all_indicators
  registry.py      # 指标注册表 (30 个指标, 6 大类)
  signal_builder.py# 信号构建辅助
```

**indicators.py** 基于 pandas-ta 实现指标计算：

- `_compute_indicator()`: 计算单个指标，返回带有命名列的 DataFrame
- `compute_all_indicators()`: 批量预计算所有默认指标参数组合，将指标列追加到原始 OHLCV DataFrame

**registry.py** 定义指标注册表，包含 30 个指标，分为 6 大类：

| 类别     | 指标                                                           |
|--------|---------------------------------------------------------------|
| trend  | EMA, SMA, WMA, DEMA, TEMA                                    |
| momentum| RSI, MACD, Stochastic, CCI, ROC, Williams %R, CMO, TRIX      |
| volatility| BB, ATR, Keltner, Donchian                                  |
| volume | OBV, CMF, MFI, RVOL, VROC, AD, CVD, VWMA                     |
| trend  | ADX, PSAR, Aroon                                              |
| guard  | VolumeProfile (仅作 guard 使用，按需计算)                       |

每个指标定义包含：参数范围（ParamDef）、输出字段、支持的条件类型、是否仅用于 guard。

#### 2.2.3 core/evolution -- 进化引擎

```
core/evolution/
  engine.py       # EvolutionEngine 主循环 + EarlyStopChecker
  operators.py    # 遗传算子 (变异 + 交叉)
  population.py   # 种群初始化与随机 DNA 生成
  diversity.py    # 种群多样性维护
  lineage.py      # 血缘追踪
```

**engine.py** 实现进化主循环：

- `EvolutionEngine.evolve()`: 接收祖先 DNA、评估函数、代回调，执行完整的进化过程
- 每代流程：评估所有个体 -> 排序 -> 精英保留 -> 交叉 -> 变异 -> 注入新鲜血液 -> 多样性维护
- `EarlyStopChecker`: 4 种停止规则（目标达成、停滞、衰退、最大代数）

**operators.py** 实现 6 种变异算子 + 交叉：

| 算子                    | 作用                                           |
|-----------------------|----------------------------------------------|
| mutate_params         | 调整指标参数（在合法范围内随机扰动）                |
| mutate_indicator      | 替换为同类别的其他指标                            |
| mutate_logic          | 翻转 AND/OR 逻辑                              |
| mutate_risk           | 调整风控参数（stop_loss, position_size, take_profit）|
| mutate_add_signal     | 添加随机 guard 信号基因                         |
| mutate_remove_signal  | 移除随机 guard 信号基因                         |
| mutate_cross_logic    | 翻转跨层逻辑（MTF 专用）                         |
| crossover             | 双亲交叉（entry 来自一方，exit 来自另一方）         |

MTF 扩展算子：mutate_add_layer、mutate_remove_layer、mutate_layer_timeframe。

**population.py** 负责种群初始化，`create_random_dna()` 生成完全随机但合法的 StrategyDNA，`init_population()` 以祖先为基础通过变异生成初始种群。

**diversity.py** 提供种群多样性维护：计算 DNA 指纹相似度、替换克隆个体、注入新鲜血液。

#### 2.2.4 core/backtest -- 回测引擎

```
core/backtest/
  engine.py       # BacktestEngine (vectorbt 封装)
  walk_forward.py # Walk-Forward 分析
```

**engine.py** 封装 vectorbt 的 `Portfolio.from_signals()`：

- 接收 StrategyDNA 和增强 DataFrame，通过 `dna_to_signals()` 生成 entry/exit 信号
- 配置手续费、滑点、止损/止盈、仓位大小、杠杆、方向
- 支持杠杆资金费率扣减（`_apply_funding_costs`）
- 强制清算检查（`_check_liquidation`）
- 返回 `BacktestResult` 数据类（含 equity_curve、trades_df、各项指标）

#### 2.2.5 core/scoring -- 评分系统

```
core/scoring/
  metrics.py      # 原始指标计算 (compute_metrics)
  normalizer.py   # 指标归一化到 0-100 分
  templates.py    # 评分模板定义
  scorer.py       # 组合评分器 (score_strategy)
```

评分流程：

```
equity_curve + trades
        |
  compute_metrics()  -> {annual_return, sharpe_ratio, max_drawdown, win_rate, calmar_ratio, total_trades}
        |
  normalize(dim, value) -> 0-100 分 (每个维度独立归一化)
        |
  template.weights 加权求和
        |
  trade_count_penalty (交易次数 < 10 时惩罚)
        |
  total_score (0-100)
```

4 种预置评分模板：

| 模板           | 年化收益 | Sharpe | 最大回撤 | 胜率 | Calmar | 阈值 |
|--------------|------|--------|------|------|--------|-----|
| profit_first | 0.35 | 0.25   | 0.25 | 0.15 | -      | 75  |
| steady       | 0.20 | 0.35   | 0.35 | -    | 0.10   | 80  |
| risk_first   | 0.10 | 0.30   | 0.40 | -    | 0.20   | 82  |
| custom       | 0.20 | 0.20   | 0.20 | 0.20 | 0.20   | 80  |

#### 2.2.6 core/data -- 数据管理

```
core/data/
  csv_importer.py # CSV 导入 (Binance 官方 / 通用 OHLCV)
  fetcher.py      # API 数据获取 (Binance API)
  storage.py      # Parquet 存储层 (save/load/merge)
  updater.py      # 数据增量更新
  mtf_loader.py   # 多时间框架数据加载器
```

**csv_importer.py** 支持：

- Binance 官方 CSV 格式（无表头，12 列）自动检测
- 通用 OHLCV CSV 格式（带表头）自动检测
- 时间戳精度自动识别（毫秒 / 微秒）
- 文件名自动解析 symbol 和 interval
- OHLCV 数据完整性验证
- 多文件批量导入与去重
- 三种导入模式：MERGE（合并）、REPLACE（替换）、NEW（新建）

**storage.py** 提供 Parquet 文件的原子操作：save、load、merge（去重合并）。

**mtf_loader.py** 提供无状态的数据加载工具函数：

- `find_parquet()`: 查找 Parquet 文件，支持 timeframe 别名（如 60m -> 1h）
- `load_and_prepare_df()`: 加载单时间框架数据，切片后计算所有指标
- `load_mtf_data()`: 加载多时间框架数据，返回 `{timeframe: DataFrame}` 字典

#### 2.2.7 core/validation -- 假设验证

```
core/validation/
  engine.py   # WHEN/THEN 假设验证引擎
  mtf.py      # 多时间框架验证辅助
  patterns.py # 价格形态模式检测
```

验证引擎接收 WHEN/THEN 条件对，在历史数据上评估触发事件的后续价格变化分布，计算匹配率、分位数、集中度等统计指标。支持触发器模式和 look-ahead THEN 评估。

#### 2.2.8 core/persistence -- 持久化层

```
core/persistence/
  db.py          # 基础数据库层 (evolution_task/generation_snapshot/evolution_history)
  checkpoint.py  # 检查点保存与恢复
```

**db.py** 管理 3 张核心表的 CRUD 操作，所有连接启用 WAL 模式。提供 init_db、save_task、update_task、save_snapshot、save_history 等函数。

**checkpoint.py** 支持进化过程的检查点保存与恢复，允许从最后一个快照继续进化。

#### 2.2.9 core/visualization -- 可视化

```
core/visualization/
  chart_builder.py   # 图表构建器 (统一入口)
  kline_chart.py     # K 线图
  equity_curve.py    # 权益曲线图
  generation_chart.py# 进化代际图
  quick_preview.py   # 快速预览
```

基于 plotly 构建可视化图表，在 API 层调用生成图表数据返回给前端。

## 4. 关键设计决策

### 4.1 为什么使用守护线程而非 Celery/Redis

- 项目定位为单机桌面级工具，无需分布式任务队列
- SQLite + WAL 模式足以支撑单写多读的并发需求
- 守护线程与主进程共享内存，避免了序列化开销
- 进化任务是 CPU 密集型而非 I/O 密集型，线程模型更合适

### 4.2 为什么 Parquet 而非数据库存储行情数据

- Parquet 列式存储对 OHLCV 数据压缩率极高
- pandas + PyArrow 直接读取 Parquet 的性能远优于 SQL 查询
- 行情数据是追加写入模式，不需要事务支持
- 指标预计算后追加到 DataFrame，Parquet 天然适合存储宽表

### 4.3 为什么预计算所有指标

- `compute_all_indicators()` 一次性计算所有默认参数组合的指标
- 进化过程中个体变化频繁，每次评估重新计算指标开销过大
- 增强后的 DataFrame 在同一次进化中可被所有个体复用
- 权衡：增加内存占用（约 100+ 列），但大幅减少重复计算

### 4.4 为什么进化引擎不使用 deap

- requirements.txt 包含 deap 依赖，但代码中未实际使用
- 自定义进化引擎更灵活，支持 MTF 层级结构的交叉/变异
- deap 的个体表示与 StrategyDNA 的嵌套 dataclass 结构不匹配
- 自定义引擎支持自适应变异强度（根据停滞代数动态调整）

### 4.5 数据库双文件策略

系统使用两个 SQLite 数据库文件：

- `data/quant.db`: 主数据库，存储 strategy、backtest_result、dataset_meta、schema_version
- `data/evolution.db`: 进化专用数据库，存储 evolution_task、generation_snapshot、evolution_history

分离的原因是进化任务写入频繁（每代多次），避免影响主数据库的查询性能。
