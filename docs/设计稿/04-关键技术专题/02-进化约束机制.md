# 02 - 进化约束机制 (Evolution Constraints)

## 1 概述

MyQuant 的进化约束机制定义了一组"任务级硬约束"(Task-level Hard Constraints),这些约束在进化任务创建时由用户设定,在整个进化过程中保持不变,不受遗传算子的影响。

本文档详细描述三类硬约束(杠杆、方向、时间周期组合)的定义、执行点、设计理由,以及相关代码路径。

---

## 2 设计原则

**人类负责架构决策,遗传算法负责参数搜索。**

三类硬约束(杠杆、方向、时间周期组合)属于策略的宏观架构层面,是用户基于市场认知和风控要求做出的设计决策。遗传算法应该在固定的架构框架内搜索最优的微观参数(指标选择、参数值、条件类型、逻辑组合),而不是尝试改变架构本身。

这一定位基于以下分析:

| 决策类型 | 决策者 | 示例 |
|---------|--------|------|
| 架构决策 | 人类 | 杠杆倍数、交易方向、时间周期组合 |
| 参数决策 | 遗传算法 | RSI period=14 还是 21, EMA 价格之上还是之下 |
| 搜索空间 | 混合 | 跨层级逻辑 AND/OR, 信号角色分配 |

## 3 约束一: 杠杆 (Leverage)

### 3.1 定义

- 取值范围: 1 - 10 (整数倍)
- 含义: 交易的杠杆倍数,1 = 无杠杆,10 = 10倍杠杆
- 位置: `risk_genes.leverage` 字段

### 3.2 约束机制

杠杆在以下两个层级被强制覆盖:

#### 层级一: 评估前覆盖

`api/runner.py` 的 `_evaluate_dna()` 方法在每次回测评估前强制覆盖:

```python
def _evaluate_dna(self, individual, task_row, leverage, direction):
    # 强制覆盖任务级约束
    individual.risk_genes.leverage = leverage
    ...
```

#### 层级二: 每代评估前覆盖

`core/evolution/engine.py` 的 `evolve()` 方法在每一代的评估循环前,对所有个体强制覆盖:

```python
for gen in range(1, self.max_generations + 1):
    # 每代开始前强制覆盖所有个体的约束
    for ind in population:
        ind.risk_genes.leverage = self.leverage
        ...
```

### 3.3 不可变异

`operators.py` 中的 `mutate_risk()` 函数不包含杠杆变异逻辑:

```python
def mutate_risk(dna: StrategyDNA) -> StrategyDNA:
    risk = data["risk_genes"]

    # 可变异: stop_loss
    if random.random() < 0.5:
        ...
    # 可变异: position_size
    if random.random() < 0.5:
        ...
    # 可变异: take_profit
    if random.random() < 0.3:
        ...

    # 杠杆: 无变异代码
```

对比 `mutate_risk()` 中可变异的风险参数:

| 参数 | 可变异 | 范围 | 步长 |
|------|--------|------|------|
| stop_loss | 是 | [0.005, 0.20] | 0.005 |
| position_size | 是 | [0.10, 1.0] | 0.05 |
| take_profit | 是 | > stop_loss | 0.01 |
| leverage | 否 | 固定 | - |
| direction | 否 | 固定 | - |

### 3.4 杠杆的影响路径

杠杆在回测引擎中的影响:

1. **仓位大小**: `size = position_size * leverage` (BacktestEngine._build_portfolio)
2. **资金费率**: `_apply_funding_costs()` 根据 leverage 计算杠杆仓位的资金成本
3. **清算检查**: `_check_liquidation()` 检查保证金率是否低于维持保证金

资金费率计算:
```python
RATE_PER_8H = 0.001
borrowed_ratio = (leverage - 1) / leverage
cost_rate = RATE_PER_8H * periods_per_bar * borrowed_ratio
```

清算线计算:
```python
maintenance = init_cash * (1 - 0.9 / leverage)
```

## 4 约束二: 方向 (Direction)

### 4.1 定义

- 取值: "long" | "short" | "mixed"
- 含义:
  - "long": 仅做多(买入开仓,卖出平仓)
  - "short": 仅做空(卖出开仓,买入平仓)
  - "mixed": 允许进化探索做多和做空两个方向
- 位置: `risk_genes.direction` 字段

### 4.2 约束机制

方向的约束机制根据取值不同而有区别:

#### long / short: 强制覆盖

与杠杆相同,在评估前和每代评估前强制覆盖:

```python
# runner.py
if direction != "mixed":
    individual.risk_genes.direction = direction

# engine.py
if self.direction != "mixed":
    ind.risk_genes.direction = self.direction
```

#### mixed: 不覆盖,允许自由探索

当方向为 "mixed" 时:
- 不执行覆盖操作
- 遗传算法可以自由在 "long" 和 "short" 之间探索
- `mutate_risk()` 虽然不直接变异 direction,但 crossover 时可能从父本继承不同方向
- `create_random_dna()` 在 mixed 模式下随机生成 "long" 或 "short"

### 4.3 方向在回测中的使用

`BacktestEngine._build_portfolio()` 将方向映射为 vectorbt 的 direction 参数:

```python
direction_map = {"long": 0, "short": 1}
portfolio_kwargs = dict(
    ...
    direction=direction_map.get(dna.risk_genes.direction, 0),
)
```

注意:vectorbt 的 direction 参数只接受 0(做多)或 1(做空),不直接支持混合方向。在 mixed 模式下,每个个体的 DNA 中 direction 字段仍然是 "long" 或 "short",只是进化过程可以选择探索哪个方向。

### 4.4 验证层

`core/strategy/validator.py` 中对方向的验证:

```python
if risk.direction not in ("long", "short"):
    errors.append(f"Direction must be 'long' or 'short', got '{risk.direction}'")
```

注意:验证器只接受 "long" 和 "short"。在 mixed 模式的进化过程中,每个个体的 direction 必须是 "long" 或 "short" 之一,mixed 本身不会出现在 DNA 中,而是作为进化任务的元参数存在。

## 5 约束三: 时间周期组合 (Timeframe Combination)

### 5.1 定义

- 取值: 有序字符串列表,如 ["3d", "4h", "15m"]
- 含义: 策略使用的时间周期组合,按从长到短排列
- 固定内容:
  - 层级数量 = len(timeframe_pool) - 1 (排除执行时间周期)
  - 各层级的时间周期分配
  - 执行时间周期 = 最短的时间周期

### 5.2 约束机制

时间周期组合的约束不是通过运行时覆盖实现的,而是通过**限制遗传算子的操作范围**来保证:

#### 种群初始化保证

`create_random_dna()` 在创建 DNA 时,严格按照 `timeframe_pool` 生成层级:

```python
if timeframe_pool and len(timeframe_pool) > 1:
    other_tfs = [tf for tf in timeframe_pool if tf != timeframe]
    layers = [create_random_mtf_layer(tf, symbol) for tf in other_tfs]
    dna.layers = layers
    dna._layers_explicit = True
```

这意味着初始种群的每个个体都具有完全相同的层级结构(相同数量、相同时间周期分配)。

#### 交叉操作保持结构

`crossover()` 按位置配对交叉:

```python
if a_has_layers and b_has_layers:
    child_layers = []
    for la, lb in zip(parent_a.layers, parent_b.layers):
        # 保持 la.timeframe 不变
        child_layers.append(TimeframeLayer(
            timeframe=la.timeframe,  # 不变
            signal_genes=layer_signals,  # 交叉
            logic_genes=layer_logic,  # 选择
        ))
```

由于种群中所有个体的层级结构相同,`zip()` 能完美配对,不会产生结构不一致的子代。

#### 禁止结构变异

进化引擎的变异池中不包含以下操作:

- `mutate_add_layer` -- 不在变异池中
- `mutate_remove_layer` -- 不在变异池中
- `mutate_layer_timeframe` -- 不在变异池中

变异池定义于 `engine.py`:

```python
mutation_pool = [
    mutate_params, mutate_indicator, mutate_logic, mutate_risk,
    mutate_add_signal, mutate_remove_signal,
]
if len(self.timeframe_pool) > 1:
    mutation_pool.append(mutate_cross_logic)
```

#### 新鲜血液保持结构

`inject_fresh_blood()` 和 `check_and_maintain_diversity()` 创建新个体时传递 `timeframe_pool`:

```python
fresh = create_random_dna(
    ...,
    timeframe_pool=timeframe_pool,
)
```

这确保注入的新个体也具有相同的层级结构。

### 5.3 时间周期组合的设计约束总结

| 属性 | 是否固定 | 机制 |
|------|---------|------|
| 层级数量 | 固定 | 种群初始化 + 禁止 add/remove layer |
| 各层级时间周期 | 固定 | 种群初始化 + 禁止 mutate_layer_timeframe |
| 层级内信号基因 | 可进化 | mutate_params, mutate_indicator, crossover |
| 层级内逻辑基因 | 可进化 | mutate_logic |
| 跨层级组合逻辑 | 可进化 | mutate_cross_logic |

## 6 执行点全景

### 6.1 约束执行流程

```
任务创建
  |-- 用户设定: leverage=3, direction="long", timeframe_pool=["3d", "4h", "15m"]
  |-- 存入 evolution_task 表
  |
  v
Runner._run_task()
  |-- 解析 leverage, direction, timeframe_pool
  |-- 创建 EvolutionEngine(leverage=3, direction="long", timeframe_pool=[...])
  |
  v
Engine.evolve()
  |-- init_population(timeframe_pool=[...])
  |     |-- create_random_dna(leverage=3, direction="long", timeframe_pool=[...])
  |     |     |-- 生成 signal_genes (执行层)
  |     |     |-- 生成 layers[0]("3d"), layers[1]("4h")
  |     |     |-- _layers_explicit = True
  |     |-- 变异生成其余个体 (结构保持)
  |
  |-- for gen in 1..max_generations:
  |     |
  |     |-- [约束覆盖点 1] 对所有个体强制覆盖
  |     |     for ind in population:
  |     |         ind.risk_genes.leverage = 3        # 强制
  |     |         ind.risk_genes.direction = "long"   # 强制 (非 mixed 时)
  |     |
  |     |-- [评估阶段] evaluate_fn(individual)
  |     |     |
  |     |     |-- [约束覆盖点 2] _evaluate_dna()
  |     |     |     individual.risk_genes.leverage = 3
  |     |     |     individual.risk_genes.direction = "long"
  |     |     |
  |     |     |-- 加载 MTF 数据 (timeframe_pool -> dfs_by_timeframe)
  |     |     |-- dna_to_signals(dna, df, dfs_by_timeframe)
  |     |     |-- BacktestEngine.run(dna, df, dfs_by_timeframe)
  |     |
  |     |-- 选择精英
  |     |-- 交叉 (保持层级结构)
  |     |-- 变异 (不触及 leverage/direction/layer 结构)
  |     |-- 新鲜血液注入 (保持层级结构)
  |     |-- 多样性维护 (保持层级结构)
```

### 6.2 双重覆盖的必要性

约束在两个位置被覆盖(Engine.evolve 循环头 + Runner._evaluate_dna):

- **Engine.evolve() 循环头**: 确保评估前个体状态正确,即使变异算子意外修改了约束字段
- **Runner._evaluate_dna()**: 作为最终防线,确保回测使用正确的约束值

这种双重保险设计使得:
1. 即使某个变异算子意外修改了 leverage/direction,在评估前也会被修正
2. 即使 Engine 层的覆盖逻辑有 bug,Runner 层仍能保证正确性
3. 未来添加新的约束相关代码时,有更强的容错性

## 8 设计理由详述

### 8.1 为什么杠杆不可进化

1. **风控需求**: 杠杆直接决定爆仓风险,是资金管理的核心参数,应由用户明确控制
2. **搜索空间**: 杠杆只有 10 个离散值(1-10),进化搜索空间过小,不值得用 GA 优化
3. **评估公平性**: 不同杠杆的回测结果缺乏可比性(高杠杆天然有更高的潜在收益和风险)

### 8.2 为什么方向通常不可进化

1. **市场认知**: 交易方向反映用户对市场的判断(看涨/看跌),这是人类的分析结果
2. **策略一致性**: 一个策略在 long 和 short 上的最优参数通常不同,混合探索会增加搜索复杂度
3. **mixed 作为例外**: 当用户不确定市场方向时,"mixed" 模式允许 GA 同时探索两个方向,但这仍然是用户的主动选择

### 8.3 为什么时间周期组合不可进化

1. **架构决策**: 时间周期组合反映策略的交易风格(短线/中线/长线),是人类的设计选择
2. **搜索效率**: 时间周期组合的搜索空间巨大(排列组合),且离散化程度高,不适合连续参数优化
3. **交叉可行性**: 固定的层级结构使得交叉操作有明确的配对关系,避免无效重组
4. **数据依赖**: 不同时间周期组合需要不同的数据准备,动态改变组合会导致数据加载复杂性爆炸

## 9 约束在遗传算子中的体现

### 9.1 各算子的约束遵循情况

| 遗传算子 | leverage | direction | 层级结构 |
|---------|----------|-----------|---------|
| mutate_params | 不触及 | 不触及 | 仅修改层级内参数 |
| mutate_indicator | 不触及 | 不触及 | 仅修改层级内指标 |
| mutate_logic | 不触及 | 不触及 | 仅修改层级内逻辑 |
| mutate_risk | 不触及 | 不触及 | 无层级操作 |
| mutate_cross_logic | 不触及 | 不触及 | 仅修改 AND/OR |
| mutate_add_signal | 不触及 | 不触及 | 仅修改层级内信号 |
| mutate_remove_signal | 不触及 | 不触及 | 仅修改层级内信号 |
| crossover | 不触及 | 可能混合* | 保持结构不变 |

*crossover 的 direction 混合: crossover 从父本 A 或 B 随机选择 risk_genes,如果两个父本 direction 不同(mixed 模式下可能),子代可能继承任一方向。但每代评估前都会被覆盖回任务约束值(非 mixed 时)。

### 9.2 crossover 的风险基因处理

```python
# crossover() 中
risk = random.choice([parent_a.risk_genes, parent_b.risk_genes])
```

子代继承其中一个父本的完整 risk_genes。在非 mixed 模式下,即使两个父本的 direction 不同(理论上不应该发生),评估前的覆盖操作会修正。

## 10 验证器角色

### 10.1 validate_dna()

`core/strategy/validator.py` 中的 `validate_dna()` 在以下校验中与约束机制协作:

```python
# leverage 范围校验
if not (1 <= risk.leverage <= 10):
    errors.append(...)

# direction 枚举校验
if risk.direction not in ("long", "short"):
    errors.append(...)
```

### 10.2 初始化时的约束注入

`create_random_dna()` 直接接收 leverage 和 direction 参数:

```python
def create_random_dna(
    ...,
    leverage: int = 1,
    direction: str = "long",
) -> StrategyDNA:
    dna = StrategyDNA(
        ...,
        risk_genes=RiskGenes(..., leverage=leverage, direction=direction),
    )
```

这确保从创建之初,个体的 leverage 和 direction 就被设定为正确的约束值。后续的变异和交叉可能改变这些值,但评估前的覆盖操作会修正。
